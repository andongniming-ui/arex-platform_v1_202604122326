# AREX 平台发布前回归 SOP

> 适用对象：测试同学、发布负责人、值班开发  
> 适用项目：`arex-platform_v1`  
> 最后更新：2026-04-12

---

## 1. 目标

这份 SOP 只回答一件事：

**在版本发布前，如何用 AREX 平台做一轮标准、可复用、可留痕的回归验证。**

如果你是第一次接手平台，建议先看：

- [AREX平台操作手册.md](./AREX平台操作手册.md)
- [MANUAL.md](./MANUAL.md)

---

## 2. 适用场景

适用于以下发布前场景：

- Java `JDK8` 服务发布
- XML / JSON 接口回归
- MySQL / MyBatis 链路服务
- 新旧环境一致性验证
- Jenkins / GitLab CI 前的人工确认

---

## 3. 发布前回归总流程

```text
确认平台状态
  -> 确认目标应用配置
  -> 选择回归范围
  -> 执行回放 / 套件 / 双环境对比
  -> 查看通过率与失败分类
  -> 针对关键失败看 Diff
  -> 输出结论
  -> 决定是否允许发布
```

---

## 4. 操作步骤

### Step 1：先确认平台本身正常

进入 `数据总览`，确认：

- 最近没有明显大面积失败
- 最近的回放趋势没有异常下降
- 平台页面和接口能正常访问

如果平台自己已经不稳定，不建议直接进入发布回归。

---

### Step 2：确认目标应用配置

进入 `应用管理`，确认目标应用：

- 应用名称正确
- SSH 配置正确
- 目标端口正确
- 需要时能正常发现 PID
- Agent 状态正常
- XML 接口需要的 `xml_request_template` 已配置

如果这一步不确认，后面很多失败都只是环境配置问题。

---

### Step 3：确定回归范围

发布前不要“全量乱跑”，建议先明确本次验证范围。

常见选择方式：

- 单条关键测试用例：适合小改动
- 回放套件：适合一组核心业务路径
- 双环境对比：适合新老环境行为比较

建议优先回归：

- 登录 / 认证
- 核心交易
- 主链路查询
- 高风险 XML 接口
- 近期改动涉及的接口

---

### Step 4：执行回放

#### 方式 A：回放中心

适用于单条或少量测试用例人工验证。

操作：

1. 进入 `回放中心`
2. 选择测试用例
3. 选择目标应用
4. 设置并发、延时、环境标识
5. 检查忽略字段、差异规则、断言、性能阈值
6. 发起回放

#### 方式 B：回放套件

适用于发布前一批核心路径统一跑。

操作：

1. 进入 `回放套件`
2. 选择现有套件或新建套件
3. 确认套件里包含的测试用例
4. 确认默认目标应用
5. 执行套件

#### 方式 C：双环境对比

适用于本次发布需要比较两个环境行为是否一致。

操作：

1. 进入 `双环境对比`
2. 选择测试用例
3. 选择环境 A 和环境 B
4. 执行对比
5. 重点查看 A/B 差异

## 5. 结果判定方式

### 5.1 先看整体指标

优先关注：

- 总通过率
- 失败数
- 错误数
- 性能阈值失败数

### 5.2 再看失败分类

进入结果详情或回放历史，区分失败类型：

- 真实业务差异
- 环境问题
- 断言失败
- 性能超时
- XML 请求体缺失或不匹配

### 5.3 最后看关键 Diff

对关键接口必须进一步看：

- 原始响应
- 回放响应
- 字段级 Diff
- 是否属于可接受差异

不要只看“红了几个”，一定要判断“为什么红”。

---

## 6. 放行标准建议

### 可直接放行

满足以下条件时，可认为回归通过：

- 核心用例全部通过
- 非核心失败都能确认是可接受差异
- 没有明显环境级错误
- 性能指标未出现显著退化

### 不建议放行

出现以下任一情况时，不建议直接发布：

- 核心交易类用例失败
- XML 关键接口返回结构变化且原因不清楚
- 多个失败集中在同一业务域
- 通过率明显低于历史基线
- 失败原因还没定位清楚

## 8. 常见异常与处理建议

### 8.1 回放失败但其实是环境问题

优先检查：

- 目标服务是否已启动
- 端口是否正确
- XML 模板是否缺失
- 下游依赖是否可用

### 8.2 XML 接口失败很多

优先检查：

- 请求体是否录完整
- `Content-Type` 是否正确
- `service_id` 是否匹配模板
- 差异是否只是时间戳 / 流水号类字段

### 8.3 双环境对比大量发散

优先检查：

- 两个环境版本是否真的是可比的
- 数据初始化是否一致
- 下游依赖是否一致

---

## 9. 推荐执行顺序

发布前最推荐按下面顺序做：

```text
数据总览
  -> 应用管理
  -> 回放套件 / 回放中心
  -> 回放历史
  -> 结果详情
  -> 输出结论
```

如果涉及新旧环境一致性，再插入：

```text
双环境对比
```

如果要接门禁，再补：

```text
CI 集成
```

---

## 10. 一句话原则

**先看整体通过率，再看失败分类，最后看关键 Diff，不要只凭状态颜色决定是否发布。**
