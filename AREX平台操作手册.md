## 1. 这是什么项目

AREX 录制回放平台是一套“真实流量录制 -> 回放验证 -> 差异分析 -> 回归固化”的平台。

它解决的问题不是“接口能不能调通”，而是：

- 线上真实业务请求是什么样
- 新版本服务是否还能按原来的行为返回
- 数据库、RPC、缓存等下游调用被 mock 后，目标服务结果是否一致
- 发布前能不能用一批真实样本快速做回归

### 1.1 一句话理解

可以把它理解成一套“把真实请求样本录下来，再自动重放到目标环境”的回归测试平台。

### 1.2 这个平台主要做什么

- 管理被测应用
- 录制真实请求和响应
- 把录制样本沉淀成测试用例
- 对目标环境执行回放
- 输出通过率、差异内容、失败原因
- 做定时回放、套件回放、双环境对比、CI 集成

### 1.3 最适合什么团队

- Java 服务团队
- XML 接口较多的老系统团队
- 有 MySQL / MyBatis / RPC 子调用链路的业务系统
- 发布频率高、人工回归成本高的团队

## 2. 适用场景与能力边界

### 2.1 适用场景

- Java `JDK8` 服务回归测试
- 线上或联调环境真实请求录制
- XML 接口回放
- JSON 接口回放
- MySQL / MyBatis / Dubbo / Redis 等子调用场景
- 灰度前、发布前、版本切换前的一致性验证

### 2.2 当前平台支持什么

从代码实现看，平台明确支持以下能力：

- HTTP 录制与回放
- XML 响应按结构化方式 Diff
- XML 请求模板自动补全
- 录制条目的 `entry_type` 区分 `HTTP / DUBBO / JAVA / MYBATIS`
- 子调用 `sub_invocations` 录制与 mock 回放
- 平台元数据库可使用 `sqlite / mysql`

### 2.3 你要特别理解的边界

- 平台本身不替代业务测试，它是“真实样本驱动”的回归工具
- 如果录制阶段没有采到关键请求，后续回放就不会覆盖这些场景
- 如果目标服务依赖的下游行为和录制时差异过大，回放结果可能失败
- XML 接口若录制请求体缺失，需要配置 XML 模板，否则无法完整回放

### 2.4 对新人最重要的判断标准

遇到一个场景先问自己 3 个问题：

1. 这个请求是否已经被录制过
2. 这个录制是否已经沉淀成测试用例
3. 这次要验证的是“单环境回放”还是“双环境对比”

## 3. 系统架构与运行原理

### 3.1 总体架构图

```text
                          ┌──────────────────────────────┐
                          │         前端 Vue3            │
                          │  页面操作 / 结果查看 / 配置    │
                          └──────────────┬───────────────┘
                                         │ HTTP
                                         ▼
┌──────────────────────────────────────────────────────────────────────────────┐
│                           AREX Platform Backend                              │
│                                                                              │
│  应用管理  录制会话  测试用例  回放任务  定时任务  套件执行  双环境对比  CI 接口 │
│                                                                              │
└──────────────┬────────────────────┬───────────────────────┬──────────────────┘
               │                    │                       │
               │                    │                       │
               ▼                    ▼                       ▼
      ┌────────────────┐   ┌──────────────────┐   ┌─────────────────────────┐
      │ 平台数据库      │   │ arex-storage      │   │ 目标 Java 服务           │
      │ SQLite / MySQL │   │ 录制数据存储服务   │   │ 挂载 arex-agent         │
      └────────────────┘   └────────┬─────────┘   └───────────┬─────────────┘
                                     │                         │
                                     │ 录制查询 / mock 缓存     │ 真实业务请求
                                     ▼                         ▼
                             ┌──────────────┐         ┌──────────────────────┐
                             │ MongoDB      │         │ MySQL / Dubbo / Redis │
                             │ Redis        │         │ 等下游系统            │
                             └──────────────┘         └──────────────────────┘
```

### 3.2 项目在做什么事

平台核心只做四件事：

1. 管理录制数据
2. 把录制数据组织成可复用测试用例
3. 把测试用例发到目标环境执行回放
4. 把差异结果整理出来给人看

### 3.3 录制原理

```text
业务请求进入 Java 服务
        │
        ▼
arex-agent 在 JVM 中拦截入口请求与子调用
        │
        ▼
请求 / 响应 / MyBatis / Dubbo / Redis 等数据被打包
        │
        ▼
写入 arex-storage
        │
        ▼
平台通过 API 拉取录制数据
        │
        ▼
落到平台数据库，形成 Recording
```

### 3.4 回放原理

```text
从测试用例中读取 Recording
        │
        ├─ 取 method / uri / body / contentType
        ├─ 必要时用 XML 模板自动补请求体
        ├─ 必要时装载 sub_invocation mock
        ▼
发请求到目标环境
        │
        ▼
拿到回放响应
        │
        ├─ 与原始 response_body 做 Diff
        ├─ 执行断言
        ├─ 计算通过率 / 性能阈值
        ▼
生成 Replay Result / HTML 报告 / 统计结果
```

### 3.5 双环境对比原理

```text
同一条测试用例
    ├─ 回放到环境 A
    └─ 回放到环境 B

平台同时比较：
- A 对录制原始响应的差异
- B 对录制原始响应的差异
- A 与 B 之间的差异
```

### 3.6 XML 为什么也能支持

平台不是把 XML 当普通字符串死比，而是会先尝试把 XML 解析成结构化对象，再做字段级对比。

这意味着：

- 可以对 XML 结构做差异分析
- 可以对 XML 使用忽略字段
- 可以根据 `<service_id>` 自动匹配 XML 请求模板

### 3.7 MySQL / MyBatis 为什么能覆盖到

因为录制模型里有 `sub_invocations`，用来保存子调用数据，包含数据库、RPC 等下游信息。  
当回放启用 mock 子调用时，平台会借助 `trace_id` 和 AREX 的缓存机制，让目标服务在执行时尽量复用录制阶段的下游返回。

## 5. 核心概念先记住这几个

### 5.1 Application

被测应用。  
一个 Application 代表一个被回放、被录制、被比较的目标服务。

### 5.2 Session

录制会话。  
一次录制任务的容器，用来承接某个时间段内采到的录制数据。

### 5.3 Recording

录制条目。  
最小业务样本，通常对应一个 HTTP 请求，也可能带有子调用信息。

### 5.4 Test Case

测试用例。  
不是手工写的脚本，而是由一组 Recording 组成的可复用回放集合。

### 5.5 Replay Job

回放任务。  
把某个 Test Case 发到某个目标应用执行一次，得到一组回放结果。

### 5.6 Schedule

定时回放任务。  
按 Cron 周期执行 Replay Job。

### 5.7 Suite

回放套件。  
把多条 Test Case 组织成一批一起跑。

### 5.8 Compare Run

双环境对比任务。  
把同一批录制同时发到两个环境，观察它们的差异。

## 6. 菜单与页面说明

这一章是新人最常用的部分。  
如果你只想知道“左侧每个菜单是干什么的”，直接看这里。

### 6.1 数据总览

**页面作用**

- 看整个平台最近的回放趋势
- 看通过率、失败数、错误数
- 快速跳转到常用页面

**适合谁看**

- 刚接手系统的人
- 发布负责人
- 想看最近质量走势的人

**你会看到什么**

- 回放任务统计
- 通过率趋势图
- 每日明细
- 日期筛选、应用筛选

**它解决什么问题**

- 最近是否有明显回归
- 哪一天开始失败增多
- 某个应用近几天通过率是否下降

### 6.2 应用管理

**页面作用**

- 注册被测应用
- 配置 SSH 信息
- 管理 Agent 挂载
- 配置录制参数
- 配置回放默认参数
- 配置 XML 模板

**核心功能**

- 新增应用
- 查看应用详情
- 测试 SSH 连通性
- 自动发现 Java PID
- 自动附加 / 解除 Agent
- 配置默认忽略字段、差异规则、断言、性能阈值
- 配置 XML 请求模板

**什么时候进这个页面**

- 第一次接入新服务
- 目标服务 IP、端口、启动方式变了
- XML 接口需要补请求模板
- 想为整个应用预设统一回放规则

**新人重点理解**

应用管理是整个项目的起点。  
应用没配好，后面的录制、回放、CI 都会出问题。

### 6.3 录制中心

**页面作用**

- 管理录制会话
- 查看录制条目
- 过滤有效样本
- 导入 HAR
- 把录制加入测试用例

**核心功能**

- 按应用、会话、入口类型、时间范围筛选
- 查看录制详情
- 查看会话详情
- 批量选中录制
- 批量加标签
- 批量加入测试用例
- 批量删除
- 导入浏览器或代理抓到的 HAR 文件

**适用场景**

- 线上刚录完流量，准备筛样本
- 没法直接挂 Agent，但手里有 HAR 文件
- 想从大量录制里挑选出稳定的关键接口

**新人重点理解**

录制中心不是最终结果页，它更像“样本池”。  
这里的录制要经过筛选和沉淀，才会进入测试用例库。

### 6.4 录制详情 / 会话详情

这两个不是左侧一级菜单，但很重要。

**录制详情**

- 查看请求体
- 查看响应体
- 看入口类型、TraceId、标签、耗时
- 查看子调用信息

**会话详情**

- 查看一个会话下所有录制
- 看这个会话覆盖了哪些接口类型
- 看平均耗时、录制数等摘要

### 6.5 测试用例库

**页面作用**

- 把录制样本沉淀成可重复执行的测试用例
- 管理用例元信息
- 查看用例包含哪些录制
- 编辑用例内录制请求

**核心功能**

- 新建测试用例
- 绑定录制条目
- 克隆测试用例
- 编辑请求参数
- 智能推荐忽略字段
- 从用例直接发起回放

**适用场景**

- 想把一次录制结果变成长期可复用回归集
- 一条录制需要修正请求体
- 想复制一个已有用例做变种

**新人重点理解**

测试用例库是“回归资产库”。  
录制是原材料，用例才是长期资产。

### 6.6 回放中心

**页面作用**

- 对某个测试用例发起一次回放
- 配置本次回放参数
- 观察任务是否启动成功

**核心功能**

- 选择测试用例
- 选择目标应用
- 设置并发、延时、环境标识
- 设置忽略字段、差异规则、断言、性能阈值
- 配置 Webhook
- 配置请求头转换
- 选择是否启用子调用 mock

**适用场景**

- 联调完成后想回归一次
- 发布前验证目标环境
- 定位某个接口是否已经偏离原行为

**新人重点理解**

回放中心关注的是“怎么跑”，不是“跑完怎么看”。  
结果分析主要在历史页和结果页完成。

### 6.7 回放历史

**页面作用**

- 查看已经执行过的回放任务
- 根据任务维度过滤历史结果
- 进入结果详情页

**核心功能**

- 看任务状态
- 看通过数、失败数、错误数
- 进入单次回放结果详情
- 进入 HTML 报告

**适用场景**

- 想知道昨天的定时任务有没有失败
- 想追某次发布前回归任务的详情
- 想导出一份报告发给别人

### 6.8 结果详情页

结果详情不是左侧独立菜单，但它是实际排查问题时最常用的页面。

**可以看什么**

- 单条录制的原始响应与回放响应
- Diff 明细
- 失败分类
- 断言执行结果
- 性能阈值是否超标
- 入口类型和 service_id

**适合谁**

- 定位问题的开发
- 做失败分类统计的测试
- 需要导出报告的发布负责人

### 6.9 定时回放

**页面作用**

- 创建周期性自动回放
- 做日常质量巡检

**核心功能**

- 创建 Cron 任务
- 启用 / 停用
- 手动立即执行
- 看上次执行时间和任务号

**适用场景**

- 每天早上自动回归
- 每次凌晨任务后自动巡检
- 固定版本环境的稳定性监控

### 6.10 回放套件

**页面作用**

- 把多个测试用例组合成一批
- 统一执行、统一统计

**核心功能**

- 创建套件
- 配置默认目标应用
- 一次执行一组用例
- 查看套件运行统计

**适用场景**

- 一个业务域有多条关键用例
- 发布前想批量回归
- 希望把“核心路径”打成一套标准检查包

### 6.11 双环境对比

**页面作用**

- 同一批录制同时回放到环境 A 和环境 B
- 找出两个环境行为差异

**核心功能**

- 创建双环境对比任务
- 查看 A 对录制、B 对录制、A 对 B 的差异
- 导出对比报告

**适用场景**

- 新老版本比较
- 灰度环境与基线环境比较
- PL / VT 等多套环境一致性检查

### 6.12 CI 集成

**页面作用**

- 把回放能力接入 Jenkins / GitLab CI / 脚本

**核心功能**

- 配置单次阻塞式回放参数
- 直接在页面验证参数
- 复制 cURL 命令
- 获取通过 / 失败结果

**适用场景**

- 发布流水线需要自动验收
- 想把某条关键用例做成门禁

### 6.13 平台设置

**页面作用**

- 给新人看平台配置说明和推荐使用流程
- 统一查看常用配置项与快速入口

**适用场景**

- 新人入门
- 忘记某个环境变量是什么意思
- 不清楚推荐操作顺序

## 7. 标准操作流程

这一章回答“我应该按什么步骤使用平台”。

### 7.1 新服务首次接入流程

```text
应用管理
  -> 新增应用
  -> 配 SSH / 端口 / JAR / 启动信息
  -> 测试 SSH
  -> 发现 PID
  -> 挂载 Agent
  -> 配录制参数
  -> 配 XML 模板（如需要）
```

### 7.2 第一次录制流程

```text
录制中心
  -> 创建会话
  -> 在被测系统上执行真实业务
  -> 停止会话并收集录制
  -> 筛选有效录制
  -> 给关键录制打标签
```

### 7.3 沉淀测试用例流程

```text
录制中心
  -> 选中关键录制
  -> 加入测试用例
  -> 到测试用例库检查排序与内容
  -> 必要时编辑请求体
  -> 推荐忽略字段
  -> 保存为稳定用例
```

### 7.4 执行单次回放流程

```text
测试用例库 / 回放中心
  -> 选择测试用例
  -> 选择目标应用
  -> 配并发 / 延时 / 环境
  -> 配忽略字段 / 差异规则 / 断言
  -> 发起回放
  -> 到回放历史或结果详情查看结果
```

### 7.5 发布前回归标准流程

```text
1. 先看数据总览确认最近平台是否稳定
2. 选核心测试用例或回放套件
3. 对目标版本执行回放
4. 检查通过率和失败分类
5. 对关键失败逐条看 Diff
6. 通过后再进入发布环节
```

### 7.6 多环境一致性验证流程

```text
双环境对比
  -> 选择测试用例
  -> 选择环境 A 和环境 B
  -> 执行对比
  -> 看 A/B 差异
  -> 导出报告给开发或测试确认
```

### 7.7 CI 接入流程

```text
CI 集成页
  -> 选用例和目标应用
  -> 页面先试跑一次
  -> 复制 cURL
  -> 放入 Jenkins / GitLab CI
  -> 以通过率或返回状态做门禁
```

## 10. 常见问题与注意事项

### 10.1 页面里很多录制，但回放结果很少

可能原因：

- 录制没加到测试用例
- 测试用例里只有部分录制
- 录制是无效样本，未纳入回放

### 10.2 XML 接口回放失败，提示没有请求体

优先检查：

- 录制里的 `request_body` 是否有内容
- `contentType` 是否是 `application/xml`
- 应用是否配置了 `xml_request_template`

### 10.3 回放明明发出去了，但结果差异很大

优先检查：

- 是否缺少忽略字段
- 目标环境和录制环境数据差异过大
- 子调用 mock 是否开启
- XML / JSON 结构是否发生了版本变化

### 10.4 录制数量很少

优先检查：

- `sampleRate` 是否过低
- 业务请求是否真的打到了挂 Agent 的进程
- `Arex.service.name` 是否和平台应用名一致
- 如果系统大量接口共用同一路径，例如都走 `/api/bank/service`，不要只依赖 Agent 自动录制
- 这类系统推荐把 Postman、功能测试平台或测试网关的目标地址改成平台代理录制地址
- 平台代理会按 `service_id / trand_id / transCode` 等字段逐条落库，更适合银行报文类接口

### 10.5 删除应用要谨慎

删除应用会连带删除与之关联的：

- 录制会话
- 录制条目
- 测试用例
- 回放任务结果
- 双环境对比结果

所以生产使用时，删除前先确认是否需要保留历史资产。

### 10.6 定时回放适合干什么

适合做：

- 巡检
- 固定回归
- 发布后观察

不适合替代：

- 首次联调验证
- 复杂问题定位
- 大范围人工探索测试

### 10.7 新人最容易犯的几个错误

- 直接在录制中心看完就走，没有沉淀测试用例
- 应用名和 `Arex.service.name` 不一致
- XML 接口没配模板就开始回放
- 用了错误目标应用做回放
- 没分清“回放中心”和“回放历史”
